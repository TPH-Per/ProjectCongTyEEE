## 4. Đợt 2 — Fix `POST /auth/v1/token` 500 unexpected_failure

### 4.1 Triệu chứng
- Login thử `quocphu02032005@gmail.com / Staff@2026` → 500 `unexpected_failure`.
- Toàn bộ user trong DB đều không đăng nhập được.

### 4.2 Root cause
- Custom Access Token hook (`20260625080844_auth_hook_custom_access_token.sql`) được
  tạo hôm qua nhưng:
  - `SECURITY INVOKER` thay vì `DEFINER` → SELECT trên `public.users` chạy dưới
    quyền `supabase_auth_admin` (GoTrue service role) nhưng RLS policy `users_self_read`
    dùng `auth.uid()` — **NULL** trong context hook → bị block → 500.
  - Không có `EXCEPTION` handler → bất kỳ NULL cast nào (e.g. user mới chưa có row
    trong `public.users`) cũng sập toàn bộ login.

### 4.3 Fix — migration `20260625100000_hook_exception_guard.sql`

```sql
-- TÓM TẮT thay đổi:
-- 1. SECURITY INVOKER → SECURITY DEFINER (chạy dưới quyền owner, bypass RLS).
-- 2. set search_path = public, auth (tránh search_path hijacking).
-- 3. EXCEPTION WHEN OTHERS THEN RETURN event; (login vẫn pass dù hook fail).
-- 4. grant select on public.users to supabase_auth_admin (defensive).
-- 5. Thêm app_metadata.hook_version = '2026-06-25' để verify hook fired qua JWT.
```

### 4.4 Verify

## 7. Đợt 5 — Fix check-in 400 (trigger `write_audit`)

### 7.1 Triệu chứng
```
XHR POST .../functions/v1/check-in [HTTP/2 400 1468ms]
Response: {"error":"null value in column \"branch_id\" of relation
                   \"audit_events\" violates not-null constraint"}
```

### 7.2 Root cause (sau 2 vòng debug)
**Vòng 1 (sai):** `write_audit` đọc branch_id qua `public.current_branch_id()` →
helper này phụ thuộc `request.jwt.claims` (rỗng trong Edge Function) hoặc
`auth.uid()` (NULL khi chạy với service role).

**Vòng 1 fix (cũng sai):** thêm `setJwtContext(admin, user.id, user.app_metadata)`
trong `auth.ts` → gọi `admin.rpc('set_config', { is_local: false })` trước INSERT.
→ **VẪN FAIL** vì PostgREST connection pool: mỗi HTTP request = 1 connection,
set_config trong request này không persist sang request kế tiếp (cùng client
nhưng pool rotation).

**Vòng 2 fix (mới - 1st attempt):** migration `20260625110000_write_audit_use_new_branch.sql`
→ trigger đọc `NEW.branch_id` (cột có sẵn trên mọi audited table) thay vì session helper.
→ **VẪN FAIL** vì trigger check `TG_ARGV[0] in ('table_assignments', ...)` (PLURAL)
mà trigger trên `table_assignments` thực tế được tạo với `'table_assignment'` (SINGULAR)
→ CASE trả NULL → fallback cũng NULL.

**Vòng 2 fix (đúng):** migration `20260625120000_write_audit_singular_fix.sql`
→ trigger đọc `(v_row ->> 'branch_id')::uuid` **UNCONDITIONALLY** từ NEW/OLD
(không phụ thuộc tên bảng). Mọi audited table đều có `branch_id` nên luôn có giá trị.

### 7.3 Fix cuối cùng
```sql
-- TRƯỚC (bug):
v_branch := coalesce(
  (case when TG_ARGV[0] in ('table_assignments','orders',...) -- PLURAL
        then (NEW.branch_id)::uuid end),
  public.current_branch_id()
);

-- SAU (fix):
v_row := to_jsonb(NEW);
v_branch := coalesce(
  (v_row ->> 'branch_id')::uuid,   -- luôn đọc từ row, không phụ thuộc TG_ARGV
  public.current_branch_id()
);
```

## 8.8 Đợt 8 — Redesign `AdminMenusView` theo Ishii + fix schema gaps

User yêu cầu thiết kế lại trang **"Thực Đơn & Gói Menu"** (`/admin/menus`)
theo đúng tài liệu `docs/member_status/Ishii/` (2-level hierarchy
category → subcategory → item, màu yellow/pink, package riêng biệt),
đồng thời fix lỗi Supabase trả về khi load trang.

### 8.8.1 Lỗi ban đầu

Khi mở `/admin/menus`, console trả:
```
[AdminMenusView] fetch failed:
  Object { code: "42703", message: "column package_items.is_active does not exist" }
```

User nghi ngờ schema cache hoặc migration chưa apply. Sau khi kiểm tra:

| Khảo sát | Kết quả |
|----------|---------|
| `package_items.is_active` trong `20260623000000_setup.sql:355` | ✅ ĐÃ ĐỊNH NGHĨA |
| `supabase migration list` | ✅ 14/14 migrations đã apply lên remote |
| `menu_subcategories` RLS policy | ❌ **KHÔNG CÓ** (chỉ có CREATE TABLE, không có `ENABLE ROW LEVEL SECURITY`) |
| `package_items` grants cho `authenticated` | ❌ **THIẾU** (chỉ có grant cho `packages`, không có `package_items`) |
| `menu_subcategories` grants | ❌ **THIẾU** cho cả `anon` lẫn `authenticated` |

**Root cause thật:** PostgREST trả `column X does not exist` (mã lỗi 42703)
khi (a) thật sự thiếu cột HOẶC (b) thiếu GRANT trên bảng được JOIN. Vì
`package_items` thiếu grant cho `authenticated`, query
`select('package_id, menu_item_id, menu_items!inner(branch_id)')` fail
tại JOIN trước khi đến được filter, làm message trông giống "thiếu cột".

### 8.8.2 Các fix schema — 2 migration mới

#### (a) `20260625220000_menu_subcategories_rls_grants_and_reload.sql`

- **Defensive** `ALTER TABLE public.package_items ADD COLUMN IF NOT EXISTS is_active boolean NOT NULL DEFAULT true` (idempotent, không phá data cũ).
- `CREATE INDEX IF NOT EXISTS package_items_is_active_idx ON package_items(package_id) WHERE is_active = true`.
- `ALTER TABLE public.menu_subcategories ENABLE ROW LEVEL SECURITY` + 2 policy (`branch_read`, `branch_write`) theo pattern `branch_id = current_branch_id()`.
- `GRANT SELECT ON menu_subcategories TO anon` (cho tablet menu public).
- `GRANT SELECT ON packages, package_items TO anon`.
- `GRANT SELECT, INSERT, UPDATE, DELETE ON menu_subcategories TO authenticated`.
- `GRANT SELECT, INSERT, UPDATE, DELETE ON package_items TO authenticated`.
- `NOTIFY pgrst, 'reload schema'` — ép PostgREST reload schema cache ngay
  (không cần restart container).

#### (b) `20260625230000_menu_items_sort_order.sql`

Sau khi fix lỗi (a), trang load được nhưng tiếp tục fail:
```
column menu_items.sort_order does not exist
```
→ Cột này có trên `menu_categories` và `package_items` nhưng **không có**
trên `menu_items` (xem `20260623000000_setup.sql:303-321`). Admin UI cần
sort item theo thứ tự thủ công trong từng danh mục.

Migration thêm:
- `ADD COLUMN IF NOT EXISTS sort_order integer NOT NULL DEFAULT 0`.
- `CREATE INDEX IF NOT EXISTS idx_menu_items_branch_cat_sort ON menu_items (branch_id, category_id, subcategory_id, sort_order) WHERE is_available = true`.
- Trigger `trg_menu_items_set_updated_at` (giữ `updated_at` fresh).
- `NOTIFY pgrst, 'reload schema'`.

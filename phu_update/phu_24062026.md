- **Nguyên nhân:** Trên Supabase, extension `uuid-ossp` được cài vào schema **`extensions`** (không phải `public`). Default `search_path = "$user", public` không bao gồm schema `extensions`, nên khi gọi `uuid_generate_v4()` mà không qualify schema, PostgreSQL không resolve được.

  - `gen_random_uuid()` là **PostgreSQL built-in** (từ v13), nằm trong `pg_catalog` → **luôn accessible** qua default search_path.
  - Output UUID v4 hoàn toàn tương đương `uuid_generate_v4()`.
  - Best practice hiện đại được khuyến nghị bởi Supabase docs.
- **Verify:**

### 5.2 Bug #11 — Hàm `current_user_role()` crash với anon role

**Triệu chứng:** Mọi request anon tới bảng RLS đều trả:
```json
{"code":"22P02","message":"invalid input value for enum user_role: \"anon\""}
HTTP 400
```

**Nguyên nhân:** JWT của anon request có `role: "anon"` (PostgREST role, không phải app role). Hàm `current_user_role()` làm:
```sql
(nullif(current_setting('request.jwt.claims', true), '')::jsonb->>'role')::user_role
```
→ cast `'anon'::user_role` raise SQLSTATE 22P02 trước khi `coalesce` fall through.

**Fix:** Whitelist role values trước khi cast — migration mới `20260624000000_fix_current_user_role_anon.sql`:
```sql
create or replace function public.current_user_role()
returns user_role language sql stable security definer set search_path = public, auth as $$
  select coalesce(
    case
      when (nullif(current_setting('request.jwt.claims', true), '')::jsonb->>'role')
           in ('admin','manager','reception','staff','kitchen')
      then ((nullif(current_setting('request.jwt.claims', true), '')::jsonb->>'role')::user_role)
    end,
    (select role from public.users where id = auth.uid())
  )
$$;
```
Đồng thời harden `has_role()` để check `current_user_role() is null` → return false thay vì raise.

**Trạng thái:** Migration file đã sẵn (`supabase/migrations/20260624000000_fix_current_user_role_anon.sql`), **CHƯA push** lên remote. Cần `supabase db push` hoặc paste vào SQL Editor Dashboard.

### 5.3 Bug #12 — Trigger tạo user mặc định `role='staff'`

**Triệu chứng:** Sau khi tạo `admin@test.com` qua Dashboard và login thành công, router guard redirect về `/manager/dashboard` thay vì `/admin/dashboard`. Console log cho thấy `profile.role = 'staff'`.

**Nguyên nhân:** Dashboard không set `user_metadata.role` → trigger `handle_new_auth_user()` chạy với `raw_user_meta_data->>'role' = NULL` → fallback `'staff'`. Đây là hành vi **đúng** (an toàn — admin không tự phong), nhưng cần UPDATE thủ công sau khi tạo.

**Fix:**
```sql
update public.users
set role = 'admin', full_name = 'System Admin', branch_id = null
where email = 'admin@test.com';
```

### 5.4 Bug #13 (phát hiện sau đợt 6) — `No match for {"name":"home"}`

**Triệu chứng:** Console warning `No match for {"name":"home","params":{}}` sau khi login.

**Nguyên nhân:** `LoginView.vue` và `RegisterView.vue` gọi `router.push({ name: 'home' })` nhưng router KHÔNG có route tên `home`. Chỉ có `path: '/'` redirect sang `/manager/dashboard` (không có name).

**Fix:** Tạo `src/utils/route.ts`:
```ts
export function getHomeRouteForRole(role) {
  switch (role) {
    case 'admin':     return { name: 'admin-dashboard' }
    case 'manager':   return { name: 'manager-dashboard' }
    case 'reception': return { name: 'reception-dashboard' }
    case 'staff':     return { name: 'staff-floor-plan' }
    case 'kitchen':   return { name: 'kitchen-kds' }
    default:          return { name: 'login' }
  }
}
```
LoginView + RegisterView + router guard dùng helper này.

### 5.5 Bug #14 (phát hiện sau đợt 6) — `ERR_CONNECTION_REFUSED` khi click link xác nhận email
